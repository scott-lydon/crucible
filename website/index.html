Crucible

An adversarial platform for AI Integrity: Crucible identifies cases where a model is silently wrong, verifies with independent checks the model and its makers cannot access, hardens the model in a closed loop using the training/configuration surface the makers expose, and measures its own catch rate against its own red agent armed with the verification scheme.

The idea

One platform, a single self-closing loop

Crucible is not a verifier and not just a fuzzer. The red agent attacks, an ensemble of independent oracles verifies, the blue loop hardens the target, and the platform measures itself — then re-attacks. The loop closes on itself and grades each pass.

In plain terms. The system under test produces an output (a fraud score, a snippet of code, a research answer), sealed off from the answer key. An oracle is a trustworthy independent check; the judge is the AI-opinion check, the weakest, with one vote.

In plain terms. The scoreboard and flight recorder. The headline number is the gap between how the target does on checks it was tuned against (validation) and brand-new checks it has never seen (held-out); a big gap means it learned to pass the test, not do the work. It records every step for replay, plots the curves, exports a reusable benchmark, and trips a kill switch if the catch rate falls below a preset red line.

Pillar 1 · Targets & oracles

owner: adapters, sealing, the four-oracle ensemble
Pros
  • Target-agnostic: one core, three thin adapters.
  • Four mechanisms that fail differently — a hack past one is caught by another.
  • Oracles derive from the sealed spec, never the submission; held-out instances made after submit.
  • Differential oracle uses a different model family.
Cons
  • The spec is only a proxy for intent.
  • Oracles could share blind spots — the hardest piece to get right.
  • Mitigation: the white-box adversary is rewarded for finding a shared gap.

Pillar 2 · Red search engine

owner: LLM search, catalog, white-box, fallback
Pros
  • LLM semantic search yields valid, interpretable evasions.
  • Persistent strategy memory compounds across runs.
  • White-box mode turns the catch-rate claim into a measured recall number.
Cons
  • LLM may fail constraint satisfaction → degrade to the hybrid.
  • Per-loop cost and latency → cache, batch, small inner model.

Pillar 3 · Blue loop

owner: automated hardening & retraining
Pros
  • Automated rapid-response patch: find the hole, close it, verify.
  • Verified on held-out attacks, not the patch's own attacks.
  • Proposes features, samples, or an ensemble automatically.
Cons
  • Can overfit to known attacks or fail to converge.
  • Mitigation: measure generalization on a held-out attack set defined up front.

Pillar 4 · Measure

owner: traces, curves, dashboard, benchmark
Pros
  • A single honest number: undetected-hack rate.
  • Per-verdict audit trace with one-click replay.
  • Exported reusable benchmark; halt-certification at a red line.
Cons
  • Self-deceiving evaluation risk → freeze the eval protocol on day one.
  • Cost-per-hack depends on LLM spend.

The core bet

Trust boundaries & spec sealing

The single hardest piece — and the reason the project is worth attempting — is independent oracle generation that survives a white-box adversary. If the oracle holds when the producer knows the scheme, the catch rate is real.

Producer can see: its own input, its own output, the query_target() score.
Producer cannot see: held-out test instances, the differential second implementation, or any oracle internals. White-box mode reveals the scheme, not the instances.
In plain terms. A checker is only worth anything if it still works when the cheater knows exactly how the checking works. White-box means the attacker holds the blueprint. Think of a lock: one that only works because burglars do not know the brand is worthless, versus one that resists a burglar holding the schematics and a key mold. Crucible bets it can build the second kind, and a catch rate measured against an attacker who knows the scheme is the only one you can believe.

Data flow

One pass, end to end

The red agent probes, reasons, and submits an evasion; the oracles verify against the sealed spec; an undetected hack is logged and patched; the dashboard records it; the white-box adversary re-attacks knowing the scheme.

Design philosophy

Decisions

Using an LLM's semantic reasoning in place of gradients as the search engine is a genuinely different design philosophy from a standard adversarial-ML pipeline. Each row is a decision the architecture commits to.

DecisionWhat we choseAlternative consideredWhy
Search engineLLM semantic reasoningGradient-based adversarial attacksGradients produce illegal or meaningless samples — negative amounts, impossible timestamps, violated business rules — that tell you nothing actionable.
VerificationNon-colluding ensemble of four oraclesSingle LLM judge, or a fixed test suiteA fixed suite is something the producer learns to pass; an LLM judge shares the producer's blind spots.
Held-out testsGenerated after submission, never exposedA static held-out setA static set leaks over time and becomes gameable.
Differential oracleSecond implementation from a different model familySecond model from the same familySame family shares blind spots, defeating the cross-check.
Target couplingThin adapter, one shared coreCore rewritten per targetKeeps the platform target-agnostic: one core, three adapters.
Catch-rate claimWhite-box adversary self-testAssert the catch rateConverts “we catch cheats” into a measured recall number against an adversary that knows the scheme.
Hard constraintsHybrid: LLM proposes, constrained search executesPure LLM searchGraceful degradation when constraint satisfaction fails, instead of failing outright.
Patch validationA held-out attack set defined up frontThe attacks used to build the patchPrevents the blue loop from overfitting to known attacks.
Stop ruleHalt certification at a residual red lineShip regardless of residualCapability-threshold gating in the spirit of a Responsible Scaling Policy.

Honest limits

Trade-offs

The deep limits are not solved. For each, Crucible shrinks the gap with a concrete mechanism, reports the residual as a number, and halts certification when that residual crosses a red line. Measuring the limits honestly is the artifact.

Per-loop LLM cost & latency

Why we accept itCache, batch, and run a small model on the inner loop.
When it would biteMany co-evolution rounds at scale, where inner-loop spend dominates.

The spec is a proxy for intent

Why we accept itReport verifier recall on the seeded corpus continuously; halt at the red line.
When it would biteUnknown hack classes exist and oversight erodes as the producer outpaces the verifier.

Oracles may share blind spots

Why we accept itFour mechanisms fail differently, and the white-box adversary is rewarded for finding a shared gap.
When it would biteAll four correlate on a class no one anticipated.

Blue overfits or doesn't converge

Why we accept itMeasure generalization on a held-out attack set defined up front.
When it would biteThe held-out set isn't representative of real attacks.

Self-deceiving evaluation

Why we accept itFreeze the evaluation protocol on day one.
When it would biteThe frozen protocol itself carries a flaw.

Constraint satisfaction fails

Why we accept itDegrade to the hybrid: the LLM proposes the strategy, a small constrained search executes it numerically.
When it would biteDomains where even a constrained numerical search can't find a valid sample.

What Crucible reports

The honest dashboard

These are the metrics the platform measures — listed, not valued. They populate from real runs at demo time; no number here is fabricated or placeheld.

Validation − held-out gap
The headline metric — proxy reward R minus ground truth R*, the slack a producer learns to exploit.
Undetected-hack rate
How often a cheat still gets through, as a single number.
Verifier recall
Catch rate on the seeded hack corpus, tracked continuously against the red line.
Cost per caught hack
LLM spend amortized over confirmed catches.
Human minutes / 1k units
Review effort per thousand units of output.
Attack & detection curves
ASR and detection rate across rounds; red↔blue co-evolution converging or oscillating.
A cost-vs-scale chart is intentionally omitted: the proposal commits no figures, and this project's standing rule is to never render placeholder or synthetic values. The chart lands once a real run produces the numbers.
